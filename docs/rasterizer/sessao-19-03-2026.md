## 2. Diagnóstico do Z-Buffer — Causa Raiz

### 2.1 Problema

O hit-map único pintava todas as 214 regiões ordenadas por level crescente. Pixels de `ci` (level 3) sobrescreviam pixels de `ps` (level 0) nas áreas sobrepostas. Uma vez sobrescrito, o dado original é destruído — nenhum filtro de leitura consegue recuperá-lo.

### 2.2 Solução Consensual (Claude + Gemini)

4 OffscreenCanvas independentes — um por nível hierárquico:

```
layer 0 → apenas regiões ps (level 0)
layer 1 → apenas regiões lf (level 1)
layer 2 → apenas regiões ph (level 2)
layer 3 → apenas regiões ci (level 3)
```

**Custo de memória:** ~2MB por layer × 4 = ~8MB — trivial.  
**Custo de runtime:** zero rebuild durante navegação.  
**Hit-testing:** `targetLevel = nav.level + 1` seleciona a layer correta.

### 2.3 Implementação

```typescript
// Build: 4 layers independentes
for (let level = 0; level <= 3; level++) {
  const canvas = document.createElement('canvas');
  // ... pintar apenas regiões do nível correspondente
  this.layers.set(level, { canvas, ctx, colorIndex: new Map() });
}

// Runtime: selecionar layer pelo nível atual
const targetLevel = nav.level + 1;
const layer = this.layers.get(targetLevel);
```

## 3. Descoberta Crítica — Estrutura Real dos SVGs

### 3.1 Análise dos 4 SVGs do Jean

Análise linha por linha dos IDs de todos os SVGs revelou duas descobertas que invalidavam premissas anteriores.

**Descoberta 1 — Slugs diferentes em cada nível:**

```
Salmon SVG:
  ps:  industrial_slaughter_process_with_tanker_truck--ps
  lf:  market_salmon--lf
  ph:  Crowding_with_sweep_nets--ph, bleeding_exsanguination--ph
  ci:  salmon--ci, well_boat--ci, pulled_net_pen--ci
```

Cada nível usa slug completamente diferente. O filtro de slug era fundamentalmente errado — nunca funcionaria.

**Descoberta 2 — Padrão de IDs inconsistente:**

```
conventional_intensive.svg usa:
  conventional--intensive_ps  ← separador underscore antes do alias
  market--pig_lf
  farrowing--crate_ci
```

O regex `/^.+--(?:ps|lf|ph|ci)(?:-\d+)?$/i` não reconhecia estes IDs.

### 3.2 Correções no Parser

**Regex expandido:**
```typescript
// ANTES:
/^.+--(?:ps|lf|ph|ci)(?:-\d+)?$/i

// DEPOIS:
/^.+(?:--|_)(ps|lf|ph|ci)(?:[_-]\d+)?$/i
```

**getLevelIndex e getAlias atualizados:**
```typescript
export function getLevelIndex(id: string): number {
  const match = id.match(/(?:--|_)(ps|lf|ph|ci)(?:[_-]\d+)?$/i);
  const alias = match?.[1]?.toLowerCase();
  return { ps: 0, lf: 1, ph: 2, ci: 3 }[alias ?? ''] ?? -1;
}
```

## 4. Filtro por parentId — Solução Definitiva

### 4.1 Por que o filtro de slug falhou

O filtro de slug assumia que ps/lf/ph/ci de um mesmo processograma compartilhavam o mesmo slug. Os dados reais provam que não — cada nível usa slug independente.

### 4.2 Implementação do filtro por parentId

```typescript
// Root: aceitar apenas regiões raiz (sem pai)
if (nav.level === -1) {
  if (region.parentId !== null) return null;
}
// Focado: aceitar apenas filhos diretos do elemento focado
else if (nav.focusedId !== null) {
  if (region.parentId !== nav.focusedId) return null;
}
```

**Por que funciona:** usa a hierarquia DOM real do SVG — independente de slug, nomenclatura ou convenção do designer.

## 5. SVG sem ps — Suporte a Hierarquias Parciais

### 5.1 Descoberta

`conventional_intensive.svg` (SVG legado, não feito pelo Jean) começa no nível `lf` — sem ps:

```
sow_lf (level 1, parentId: null)
  └── farrowing_ph (level 2)
  └── gestation_ph (level 2)
  └── prepuberty_ph (level 2)

market--pig_lf (level 1, parentId: null)
  └── growing_ph (level 2)
  └── postweaning_ph (level 2)
  └── suckling_ph (level 2)
```

A layer 0 tinha zero regiões — clique no root retornava null.

### 5.2 Correção — Root varre todas as layers

```typescript
if (nav.level === -1) {
  // Varrer layers 0→3 em ordem crescente
  // Retornar primeira região com parentId === null
  for (let lvl = 0; lvl <= 3; lvl++) {
    const layer = this.layers.get(lvl);
    if (!layer || layer.colorIndex.size === 0) continue;
    // ... ler pixel ...
    if (region.parentId !== null) continue;
    return regionId;
  }
  return null;
}
```

**Resultado:** suporta SVGs que começam em qualquer nível hierárquico.

### 5.3 targetLevel baseado no level real

```typescript
// ANTES (problemático para SVGs sem ps):
const targetLevel = nav.level + 1;

// DEPOIS (correto):
const focusedRegion = nav.focusedId
  ? this.regionMap.get(nav.focusedId)
  : null;
const targetLevel = focusedRegion
  ? focusedRegion.level + 1
  : nav.level + 1;
```

## 6. Correção de Bbox — getCTM() e Problema de Escala

### 6.1 Sintomas observados

- **Sintoma 1:** Câmera zoomava para posição errada (coordenadas distorcidas)
- **Sintoma 2:** Crash `Cannot read properties of undefined (reading 'x')`

### 6.2 Introdução do getCTM()

Primeira tentativa: usar `getCTM()` para obter coordenadas absolutas:

```typescript
const ctm = (el as SVGGraphicsElement).getCTM?.();
// Transformar 4 cantos do bbox pelo CTM
const corners = [...].map(p => {
  const pt = svgEl.createSVGPoint();
  return pt.matrixTransform(ctm);
});
```

### 6.3 Problema do getCTM() — Escala do Viewport

O `getCTM()` retorna a matriz em relação ao **viewport renderizado**, não ao viewBox matemático. O container temporário (`tempContainer`) não tem dimensões CSS definidas, então o browser aplica scale-to-fit — corrompendo a matriz com a escala do viewport.

**Resultado:** `globalBbox` em pixels de tela do container fantasma, não em unidades do viewBox.

### 6.4 Solução Proposta (pendente de implementação)

Proposta consensual Claude + Gemini:

1. Remover o bloco `getCTM()` completamente
2. Normalizar o SVG root antes do loop:

```typescript
// Forçar width/height = valores do viewBox
// Garante renderização 1:1 — getBBox() retorna unidades puras do viewBox
const vb = svgRoot.viewBox?.baseVal;
if (vb && vb.width > 0) {
  svgRoot.setAttribute('width',  String(vb.width));
  svgRoot.setAttribute('height', String(vb.height));
}
```

3. Usar `getBBox()` puro — que já opera no espaço do viewBox nativamente

**Justificativa:** `getBBox()` retorna coordenadas no sistema de coordenadas do usuário do SVG (espaço do viewBox) — não do viewport. O `getCTM()` foi introduzido para resolver um problema que `getBBox()` já resolvia nativamente.
